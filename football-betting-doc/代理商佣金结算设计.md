# 代理商佣金与结算设计（文档稿，不含 SQL）

本文档描述「拉新佣金 + 充值佣金」的产生规则、佣金支付明细模型、批量支付与校验、快照与幂等，以及管理端统一列表 UI。**暂不定义退款 / 冲正策略**（见第 8 节）。**本文不出现建表 SQL**，仅做产品与数据结构设计说明。

---

## 1. 佣金产生（两种来源）

| 来源 | 触发条件 | 台账行为 |
|------|----------|----------|
| 拉新 | 用户完成注册（且渠道归属某代理商、业务规则认定有效） | 为该用户、该代理商**新增一行**「佣金支付明细」，类型为**拉新** |
| 充值 | 用户充值订单在平台侧**支付成功**（如 `payment_orders` 状态为已付） | 为该笔订单、该用户、该代理商**新增一行**「佣金支付明细」，类型为**充值** |

原则：**一行明细 = 一次可计费事件**；应付金额在入账时已根据当时规则算好并落库（快照），后续不随配置漂移回算。

---

## 2. 佣金支付明细（逻辑字段）

表名实现阶段可自定（如 `agent_commission_lines`）。以下按**逻辑字段**列出。

### 2.1 主键与归属

| 字段 | 说明 |
|------|------|
| `id` | 主键 |
| `agent_id` | 代理商 |
| `user_id` | C 端用户 |
| `username` | 展示名**快照**（入账时写入，避免改名影响对账） |

### 2.2 类型与时间

| 字段 | 说明 |
|------|------|
| `commission_type` | 枚举：`registration`（拉新）/ `recharge`（充值） |
| `created_at` | 佣金产生时间（入账时间） |

### 2.3 拉新相关（仅拉新行有业务意义）

| 字段 | 说明 |
|------|------|
| `reg_factor` | **拉新系数快照**：产生该行时适用的系数（业务上可理解为「折算到业绩 / 计价口径」的系数；与货币单位命名解耦，历史上若称「元/人」可仍用同一数值，字段名统一为 `reg_factor`） |

充值行：`reg_factor` 可为 `NULL`。

### 2.4 充值相关（仅充值行有业务意义）

| 字段 | 说明 |
|------|------|
| `payment_order_id` | 关联平台充值订单（主键或业务单号，实现二选一或并存需统一真相源） |
| `recharge_amount` | 充值金额快照 |
| `rebate_rate` | **返点率快照**（小数或百分比按实现约定，需与展示一致） |

拉新行：上述字段可为 `NULL`。

### 2.5 金额与支付

| 字段 | 说明 |
|------|------|
| `commission_amount`（或 `commission`） | **应付佣金**（与线下打款币种/单位一致，建议元，小数位统一） |
| `paid_at` | 支付完成时间；未支付为 `NULL` |
| `admin_id` | 经办管理员 |
| `payout_rorder_id` | 线下打款凭证号（用户所述「支付佣金订单号」） |
| `payment_status` | 如：`pending` / `paid` |
| `payment_batch_id` | 同一批次勾选多行打款时共用，便于对账与审计 |

---

## 3. 快照原则

- 入账时写入的 `reg_factor`、`recharge_amount`、`rebate_rate`、`commission_amount`、`username` 等均为**事件发生时**的规则与展示快照。
- 之后代理商档案中的默认费率、档位表、活动配置变更**不反向修改**已存在明细行。
- 与历史「按月汇总再结算」公式可并存；**财务对账以明细行为准**。

---

## 4. 幂等约束（建议）

避免同一业务事件重复计佣：

| 类型 | 建议唯一性（业务键） |
|------|----------------------|
| 拉新 | 同一 `agent_id` + 同一 `user_id` + 类型拉新，在「只允许一笔有效拉新佣」的产品规则下可建唯一约束；若产品允许多笔需另行说明 |
| 充值 | 同一 `agent_id` + 同一平台订单（如 `payment_order_id` 或 `recharge_order_id`）唯一 |

支付回调、注册回调重复触发时，第二次应**不产生新行**或走明确错误处理。

## 5. 批量支付与金额分散（你的例子）

**不要求**每笔产生后立即线下打款。管理员可勾选同一代理商下**多笔**状态为「待支付」的明细，填：

- 本次**实付总金额**（须校验）
- `payout_reference`（凭证号）

保存后：对**每一条被选中的明细**写入相同的 `paid_at`、`admin_id`、`payout_reference`、`payment_status = paid`，以及相同的 `payment_batch_id`。

**金额分散的含义**：各行的 **`commission_amount` 在入账时已确定**（如 1 元与 8 元）；批量支付**不是**把 9 元重新拆成未知比例摊到各行，而是**实付 9 元 = 所选行 `commission_amount` 之和**。若不相等，应拒绝保存或要求产品定义「部分支付 / 尾差」规则（当前文档建议：**先强制相等**）。

**校验（建议强制）**：

\[
\sum_{\text{所选行}} \texttt{commission\_amount} = \texttt{本次实付金额}
\]

且仅允许勾选 `payment_status = pending` 的行（或等价状态）。

---

## 6. 与平台订单的关系

- 充值类明细必须在订单**支付成功**后生成，并与平台 `payment_orders`（或等价表）可关联追溯。
- 拉新类明细在注册成功且规则满足时生成，与 `users.agent_id` 等归因字段一致。

---

## 7. 管理端 UI：统一「佣金明细」

不再区分「拉新列表」「充值列表」两套结算表；**一张列表**展示所有佣金明细（可按代理商筛选 / 时间筛选）。

**列建议**：

| 列 | 说明 |
|----|------|
| 用户名 | `username` 快照 |
| 佣金类型 | 拉新 / 充值 |
| 佣金 | `commission_amount` |
| 备注 | 由程序生成：`reg_factor=…` 或 `充值金额 + 返点率` 等可读说明 |
| 日期时间 | `created_at` |
| 支付状态 | `payment_status` |
| 支付日期时间 | `paid_at` |

操作：勾选多行 → 结算 → 录入实付金额与凭证号 → 保存（触发批量校验与更新）。

## 9. 文档示例（与业务叙述对齐）

代理商 A：

- 用户 1 拉新：一行，`commission_type=registration`，`reg_factor` 与规则一致，`commission_amount=1`。
- 用户 2 充值 100：一行，`commission_type=recharge`，`recharge_amount=100`，`rebate_rate` 快照，`commission_amount=8`。

管理员勾选两行，线下转账 9 元，凭证号记 `payout_reference`，保存后两行均 `paid`，共用 `payment_batch_id`，且 `1+8=9` 校验通过。
